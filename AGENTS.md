# Repository Guidelines

## Project Structure & Module Organization

This repository contains a Vim plugin for browsing GitHub repositories without cloning them.

- `plugin/gb.vim` defines user commands, defaults, and autocommands.
- `autoload/gb.vim` contains public entry points used by commands and autocommands.
- `autoload/gb/*.vim` contains focused implementation modules:
  - `path.vim`: input parsing, canonical path formatting, GitHub URL generation.
  - `github.vim`: GitHub REST transport through `curl`.
  - `cache.vim`: tree, content, branch, search, and default-branch caches.
  - `browser.vim`: remote directory browser buffer.
  - `completion.vim`: `:GBrowse` completion.
- `doc/gb.txt` is the Vim help manual. `doc/tags` is generated by `:helptags`.
- `test/smoke.vim` is an offline parser smoke test.
- `test/integration.vim` is a networked GitHub API smoke test.
- `test/layout.vim` is a networked browser window-layout smoke test.

## Build, Test, and Development Commands

Run syntax/loading and parser checks:

```sh
vim -Nu NONE -i NONE -n -es -S test/smoke.vim
```

Run the networked integration smoke test:

```sh
vim -Nu NONE -i NONE -n -es -S test/integration.vim
```

Run the browser layout smoke test:

```sh
vim -Nu NONE -i NONE -n -es -S test/layout.vim
```

Regenerate help tags after editing `doc/gb.txt`:

```sh
vim -Nu NONE -i NONE -n -es -c 'set rtp^=.' -c 'helptags doc' -c 'qa!'
```

## Coding Style & Naming Conventions

Use Vimscript compatible with Vim 8.2+. Prefer small autoload functions with names matching their module, for example `gb#path#parse_input()` and `gb#cache#ensure_tree()`. Use two-space indentation and keep script-local state in `s:` variables. Public configuration should use `g:gb_*` names and be documented in `doc/gb.txt`.

Avoid adding dependencies beyond standard Vim features, `curl`, and optional `gh` token lookup.

## Testing Guidelines

Add parser and pure-function coverage to `test/smoke.vim`. Add GitHub API behavior checks to `test/integration.vim`, using stable public repositories and explicit refs. Tests should fail through Vim’s `assert_*` functions and `v:errors`.

Run both tests before submitting changes that touch parsing, transport, cache behavior, or browser opening.

## Commit & Pull Request Guidelines

No project Git history is available in this workspace, so use conventional, imperative commit subjects such as `Add branch completion cache` or `Fix canonical path parsing`.

Pull requests should include a short behavior summary, affected commands or modules, test commands run, and any user-facing documentation updates. For browser UI changes, include a brief before/after description.

## Security & Configuration Tips

Do not commit tokens. Prefer `$GITHUB_TOKEN` or `gh auth login` over setting `g:gb_github_token` in a shared config. Keep remote buffers read-only unless write support is explicitly designed and documented.
